# Crumble Rough.js Implementation Guide

This document explains the Rough.js architecture in this project from the perspective of a senior frontend engineer onboarding a junior developer.

Scope of this guide:

- Rough.js runtime configuration
- Theme and variant option resolution
- Seed strategy (stable vs random)
- Context and app-level wiring
- Shared hook behavior
- Primitive components
- UI components
- Registry/export structure
- Practical gotchas and improvement opportunities

Out of scope:

- Fumadocs internals
- MDX content authoring
- shadcn CLI mechanics

---

## 1) Mental model in one minute

Crumble draws hand-drawn UI by rendering SVG shapes generated by Rough.js.

At runtime, each component follows a similar pipeline:

1. Read global config (theme + animation flags)
2. Resolve Rough options via theme + variant + local overrides
3. Choose a seed (stable for deterministic strokes, random for redraw jitter)
4. Draw one or more Rough-generated SVG groups into an overlay SVG
5. Redraw on relevant triggers (mount, hover, focus, value change, resize)

The visual style is therefore data-driven through theme presets and variant overrides, not hardcoded per component.

## 2) Core configuration layer

### 2.1 `lib/rough.ts`

This is the core source of truth.

#### Themes

`CrumbleTheme` supports:

- `pencil`
- `ink`
- `crayon`

Each theme maps to a Rough.js `Options` object in `roughThemes`.

High-level personality:

- `pencil`: medium roughness, thin stroke, hachure fill style
- `ink`: lower roughness, thicker clean stroke, solid fill
- `crayon`: high roughness, thick stroke, zigzag fill texture

#### Component variants

`ComponentVariant` supports:

- `border`
- `fill`
- `interactive`
- `chart`

`componentOverrides` applies per-variant tuning on top of the theme defaults.

Example reasoning:

- interactive elements get more wobble than static borders
- fill-heavy components adjust `fillWeight` and `hachureGap`

#### Option resolver

`getRoughOptions(theme, variant, extra)` merges in this order:

1. theme base options
2. variant override options
3. component-local extra options

Later layers win.

#### Seeds

- `stableSeed(id)` gives deterministic output from a string hash
- `randomSeed()` creates per-draw randomness

Why this matters:

- deterministic seeds avoid visual flicker across rerenders
- random seeds are used intentionally on hover/interaction for "alive" sketch effect

#### Mutable global config

`config` stores:

- `theme`
- `animateOnMount`
- `animateOnHover`

`configureCrumble` mutates this singleton.
`getCrumbleConfig` returns a shallow copy.

This is simple, but it is module-global state.

---

### 2.2 `lib/crumble-context.tsx`

`CrumbleProvider` bridges React world to that module-global config.

Behavior:

- keeps current theme in React state
- calls `configureCrumble` whenever provider props/state change
- exposes context value with `setTheme`
- wraps children in a `div` with `data-crumble-theme=<theme>`

The data attribute is used by CSS tokens and can be used for theme-based selectors.

Important detail:

- The context and the mutable singleton both exist.
- Most components read the singleton (`getCrumbleConfig`) directly, not context.

So context mainly controls and synchronizes global settings, rather than being consumed everywhere.

---

### 2.3 App-level wiring (`app/layout.tsx`, `app/global.css`)

`app/layout.tsx`:

- wraps the app with `CrumbleProvider theme="pencil"`
- defines SVG filters in `<defs>` for pencil/ink/crayon wobble distortion

`app/global.css`:

- defines CSS tokens per `[data-crumble-theme]` (`--cr-weight`, `--cr-sketch-duration`)
- defines `.crumble-border[data-crumble-theme=...]` filter rules targeting the SVG filters

Observation:

- None of the current Rough components appear to apply `.crumble-border` directly.
- So the filter rules are prepared, but may be underused unless class names are applied externally.

## 3) Shared hook layer

### 3.1 `hooks/use-rough.ts`

This is the abstraction that most components use.

Inputs:

- `variant`
- `options` (extra rough options)
- `stableId`
- `theme` override
- optional external `svgRef`

What it returns:

- `svgRef`
- renderer ref (`rc`)
- helper draw functions (`drawRect`, `drawCircle`, `drawLine`, `drawPath`)
- `getOptions`
- resolved `theme`

Renderer lifecycle:

- lazily creates Rough SVG renderer via `rough.svg(svgElement)`
- caches renderer and recreates only when SVG element reference changes

Seed strategy inside `getOptions`:

- explicit `extra.seed` wins
- else if `stableId` exists -> deterministic `stableSeed(stableId)`
- else if hover animation is enabled -> random seed
- else fallback deterministic seed

Design benefit:

- components can request redraw randomness by just passing `seed: randomSeed()`

### 3.2 Registry wrappers

`registry/new-york/hooks/use-rough.ts` re-exports root hook.
`registry/new-york/lib/rough.ts` and `registry/new-york/lib/styles.ts` re-export root libs.

Purpose:

- keep registry import paths stable for published component consumers
- avoid duplicating implementation logic

## 4) Primitive components

These are low-level rough drawing primitives.

### 4.1 `rough-rect.tsx`

- Uses `useRough(..., variant: "border")`
- Measures container size and draws rectangle inset by 2px
- Optional hover reseed via `randomSeed()`
- Optional `rounded` maps to lower roughness for smoother corners
- Uses `ResizeObserver` to redraw on size changes

### 4.2 `rough-circle.tsx`

- Draws circle centered in given diameter
- Hover reseed supported
- Can wrap child content in centered overlay

### 4.3 `rough-line.tsx`

- Horizontal or vertical line mode
- Uses fixed 20px cross-axis size and dynamic main axis
- Redraws on resize

### 4.4 `rough-path.tsx`

- Accepts arbitrary SVG path `d`
- Allows option overrides via hook `options`
- Hover reseed supported

### 4.5 `rough-arrow.tsx`

- Draws curved quadratic path from `(x1,y1)` to `(x2,y2)`
- Computes arrowhead geometry from tangent angle
- Arrowhead size is theme-aware (`crayon` larger)
- No hover reseed by default; redraw mainly theme/prop-driven

### 4.6 `rough-highlight.tsx`

- This one does not use `useRough`; it creates `rough.svg(svg)` directly
- Supports annotation styles:
  - underline
  - box
  - circle
  - highlight block
  - strike-through
  - bracket
- Supports mount animation by animating path stroke dash offset
- Duration defaults by theme (ink fastest, pencil slowest)
- Uses `ResizeObserver` to keep annotation aligned with content

Why direct Rough usage here?

- It needs custom multi-node choreography and path-level animation control.

## 5) UI components (Rough applied to controls)

### 5.1 `button.tsx`

- Draws a rough rectangle around button bounds
- Uses `interactive` variant for more lively wobble
- Redraw triggers:
  - mount/theme change
  - resize
  - mouse down (forced reseed)
  - hover enter/leave if global hover animation enabled
- Variant only changes stroke color (`default`, `ghost`, `destructive`)

### 5.2 `card.tsx`

- Draws main rough border + optional stacked paper layers
- Stacked mode draws secondary/tertiary rotated, offset low-opacity rectangles
- Hover reseed affects front border
- Resize-aware

### 5.3 `input.tsx`

- Supports `box` or `underline` style
- Stroke color reflects validation/focus/idle state
- Rough outline redraws on focus changes and resize
- Input element itself is transparent and sits above SVG

### 5.4 `textarea.tsx`

- Similar to input but always rectangular border
- Optional auto-grow adjusts textarea height and redraws border
- Focus/error stroke handling mirrors input

### 5.5 `select.tsx`

- Draws rough rectangular box + rough chevron icon with two rough lines
- Focus/error state controls stroke color
- Uses native `<select>` with `appearance-none`

### 5.6 `checkbox.tsx`

- Drawn directly with `rough.svg` (not hook)
- Draws square border, and two rough lines for checkmark when checked
- Hover can reseed for sketch jitter
- Uses stable seed by id for deterministic base state

### 5.7 `radio.tsx`

- Each item drawn directly with `rough.svg`
- Draws outer circle + inner filled circle when selected
- Hover reseed per option
- Supports disabled visual state

### 5.8 `slider.tsx`

- Drawn directly with `rough.svg`
- Track is two rough thick lines:
  - background track
  - filled progress track
- Thumb is rough circle in separate SVG
- Uses stable seeds for track/fill/thumb to keep shape identity stable
- Native range input remains invisible interaction layer

### 5.9 `toggle.tsx`

- Drawn directly with `rough.svg`
- Track is rounded-looking rough rectangle
- Thumb is rough circle; position changes with state
- Controlled/uncontrolled hybrid behavior:
  - internal state for default usage
  - `checked` prop for controlled mode

## 6) Registry and distribution structure

### 6.1 `registry.json`

Defines all publishable items, their dependencies, and source-to-target file mapping.

Important for Rough stack:

- `rough-lib` publishes core config utilities
- `styles-lib` publishes shared inline style objects
- `use-rough` publishes hook and depends on `rough-lib`
- each primitive/UI item declares Rough and registry dependencies

### 6.2 `public/r/*.json`

These are registry item payloads used by install commands like:
`npx shadcn add https://crumble.dev/r/button.json`

They embed file content and metadata for each component/lib item.

## 7) Runtime flow examples

### 7.1 Hover redraw flow (Button)

1. User hovers button
2. `onMouseEnter` calls `draw(true)`
3. `drawRect(..., { seed: randomSeed() })`
4. Rough produces a slightly different stroke shape
5. User leaves hover -> redraw with stable seed baseline

### 7.2 Theme switch flow (Preview)

1. `PreviewContainer` updates local theme state
2. It re-renders nested `CrumbleProvider theme={theme}`
3. Provider calls `configureCrumble`
4. Components that depend on returned `theme` from `useRough` rerun draw effect
5. New theme options change stroke width/roughness/fill style

## 8) Engineering trade-offs and gotchas

### 8.1 Mutable singleton config

Pros:

- very simple API
- easy external use

Cons:

- less React-idiomatic
- potential mismatch if multiple providers coexist with different themes
- can complicate SSR or concurrent rendering assumptions

### 8.2 Draw performance

- Redraw is imperative and relatively cheap for small SVGs
- `ResizeObserver` on many nodes can still add overhead in dense UIs
- Hover reseed causes frequent redraws by design

### 8.3 Controlled/uncontrolled patterns

- `checkbox`, `radio`, `slider`, `toggle` mix controlled and uncontrolled patterns differently
- Works, but consistency across components could be improved for predictability

### 8.4 CSS wobble filters

- Filter defs and CSS rules are present
- Class hook (`.crumble-border`) is not broadly applied in current components
- If the intent is global wobble post-processing, class application strategy should be standardized

## 9) How to add a new Rough component (recommended pattern)

1. Start with `useRough` unless the component needs custom multi-shape animation choreography.
2. Pick a variant intentionally:
   - static border -> `border`
   - interactive control -> `interactive`
   - filled shape -> `fill`
3. Give a meaningful stable id seed so baseline rendering is deterministic.
4. Redraw on:
   - mount + theme changes
   - resize if dimensions are dynamic
   - specific interactions (hover/focus/press) if needed
5. Keep native control (input/select/etc.) as the accessibility interaction layer, with SVG as visual layer.
6. If hover jitter is desired, pass `seed: randomSeed()` only on interactive redraw events.
